SBOM完全ガイド：
SPDX / CycloneDX / Excel運用・提出と機密の現実解

自動車サプライヤーにとって、SBOMはもはや「一部の大企業だけが対応するもの」ではありません。背景にあるのは、車両がソフトウェア集約型になり、出荷後も脆弱性や更新を継続管理する必要が高まっていることです。

ISO/SAE 21434は、車両のE/Eシステムに対するサイバーセキュリティ工学をライフサイクル全体で扱う標準として定義しており、UN Regulation No.155は車両メーカーに継続的なサイバーセキュリティ管理を求めています。日本でもJAMA/JAPIAがガイドラインを公開し、自己評価とPDCAによる改善運用を後押ししています。

つまり、SBOMに取り組む意味は、単なる海外規制対応ではなく、取引・監査・脆弱性照会に耐える管理レベルを作ることにあります。本記事では、SBOMの本質からフォーマットの選び方、Excelでの始め方、機密情報の扱い方までを徹底解説します。

1. 自動車サプライヤーがSBOMでつまずく理由

SBOMの話になると、現場ではまず次のような反応が出ます。

  • 「OSSをそんなに多く使っていない」
  • 「ライブラリ一覧なら設計担当の頭の中にある」
  • 「OEMから正式に様式を求められていない」
  • 「標準フォーマットが難しすぎる」

この感覚自体は自然です。実際、Tier2〜Tier4の現場では、専任のセキュリティ担当者やPSIRTがあるとは限りません。ですが、問題は脆弱性が出た瞬間に“誰も即答できない”状態にあることです。SBOMがないと、どの製品に何が入っているかを毎回ゼロから調べることになり、OEM照会、監査、是正計画のどれも遅れます。SBOMの本当の価値は、平時ではなく有事の初動を速くすることです。

特に自動車業界では、ECU単位、機能単位、プロジェクト単位で構成が変わり、同じコンポーネントでも設定が異なる場合があります。だからこそ、SBOMは“理想的に全部を最初から表現する”ことより、どの単位で棚卸しし、どの単位で更新するかを先に決めた方が立ち上がります。ここを曖昧にすると、SPDXだろうがCycloneDXだろうが、結局は使われないファイルになります。

2. まず押さえるべき2大フォーマット

SPDX と CycloneDX は何が違うのか

実務で最初に比較対象になるのは、SPDX と CycloneDX です。

SPDXはLinux Foundation系プロジェクトの仕様で、ISO/IEC 5962:2021 の国際標準として位置づけられています。ライセンス情報やソフトウェア部品表の標準化、文書化の面で安心感があり、「標準として説明しやすい」ことが大きな強みです。

一方のCycloneDXは、OWASP主導で発展し、v1.6がEcma International standardとなりました。単なるSBOM仕様ではなく、ソフトウェアからAI/ML、暗号まで扱えるxBOM標準として打ち出されており、脆弱性やVEXなど、セキュリティ運用との接続を意識したエコシステムが強いのが特徴です。

この違いを、自動車サプライヤー目線で言い換えるとこうです。

  • SPDX：「標準性・文書性・ライセンス整理の強さ」
  • CycloneDX：「脆弱性運用・機械可読・拡張性の強さ」

実務では、次のような見分け方が扱いやすいです。

取引先説明・標準性重視 → SPDX寄り
脆弱性突合・VEX・自動連携重視 → CycloneDX寄り
まずはExcelで始めて、後でどちらかへ移行 → 現実的

3. 最初はExcelで始めてよい

“機械可読でないと意味がない”は誤解

結論から言うと、最初はExcelで始めて構いません。

CISAはSBOMを透明性とリスク管理の基盤と説明していますが、重要なのは「最初から完璧な標準フォーマットであること」より、運用に乗ることです。現場で一番価値があるのは、CVEが出た日に「そのコンポーネントは入っている／いない」「どの製品バージョンまで関係する」と言えることです。Excelでも、それができれば最初の壁は越えています。

もちろん、最終的にはSPDXやCycloneDXへ寄せた方が自動突合に有利です。しかし、Excelでさえ整わない状態でツールだけ導入すると保守されなくなります。自動車サプライヤーの現実解は、Excelで“正しい棚卸しの単位”を作ってから標準化することです。

4. Tier2〜Tier4向け 最小SBOMに入れるべき項目

MVP的に始めるなら、SBOMには最低でも次の項目を入れてください。

  • 製品名
  • 製品バージョン
  • コンポーネント名
  • コンポーネントバージョン
  • 取得元 / 供給元
  • OSS / 商用 / 自社開発の区分
  • 搭載箇所（ECU / 機能 / モジュール）
  • 利用条件や備考（機能無効、ビルド条件など）
  • 更新日
  • 責任部門 / 更新担当者

特に見落とされやすいのが、搭載箇所 と 備考 です。同じライブラリでも、製品によって外部通信経路に乗るかどうかが異なります。この差が、後で「影響あり / なし」の判断に直結します。

5. SBOM作成の実務フロー

いきなり全部やらない

SBOMで失敗する典型例は、「全製品」「全バージョン」「全サプライヤー」を同時にやろうとして止まることです。現実的には、トップ3製品、またはOEM照会が多い製品から始めるのがよいです。

実務フローとしては、次の順番が安定します。

  1. まず、製品単位で使用ソフトウェアを棚卸しする。
  2. 次に、版数と取得元を正規化する。
  3. その後で、製品バージョンとコンポーネントを紐づける。
  4. 最後に、更新責任者と更新タイミングを決める。

6. 更新頻度と版管理

SBOMは“静的文書”ではない

最低でも、以下のタイミングでは更新すべきです。

  • 新規リリース時
  • マイナー更新 / パッチ適用時
  • 外部ライブラリ更新時
  • 機能追加や構成変更時
  • OEM提出前の確認時

版管理でおすすめなのは、製品版数とSBOM版数を分けて持つことです。たとえば製品が `2.1.0` でも、SBOM自体は `2.1.0-a`、`2.1.0-b` のように改訂される場合があります。これを分けておくと、「製品は同じだが、脆弱性情報を追記して部品表の精度が上がった」といった履歴を残しやすくなります。

7. 機密情報と提出範囲

“全部見せる”必要はない

SBOMに対する最大の懸念の1つが、設計情報の漏えいです。SBOMを作ることと、完全な情報を誰にでも配ることは同じではありません。

現実的な整理は、次の3層です。

  1. 社内完全版：詳細な搭載箇所、備考、ビルド条件まで含む
  2. OEM提出版：必要範囲の構成要素と版数を中心にする
  3. 照会回答版：対象CVEに関係する部分だけを抜粋する

この分け方をしておくと、機密を守りながら、必要な説明責任は果たしやすくなります。

8. SBOMと脆弱性管理をつなぐ

NVD / CPE / PURL の実務

SBOMが本当に効いてくるのは、NVDなどの脆弱性管理とつながった瞬間です。ここで重要になるのが「何の識別子を持つか」です。CycloneDXは、OSSにはPURLを推奨し、プロプライエタリソフトウェアやNVDとの照合にはCPEが適すると説明しています。

OSSやパッケージ系 → PURLを優先
NVD照合やOEM説明 → CPEも理解する

9. SBOMとVEXをつなげる

“影響なし”を説明できるようにする

SBOMだけでは「入っているか」は分かっても、「その脆弱性が実際に悪用可能か」は分かりません。そこでVEX（Vulnerability Exploitability eXchange）が必要になります。

CISAのVEX最小要件では、特に NOT AFFECTED（影響なし）の説明で役立つ Justification（根拠）の考え方が整理されています。「コンポーネントが存在しない」「脆弱コードが存在しない」「実行経路にない」「攻撃者が制御できない」「緩和策がある」といった語彙を使うことで、OEMへの説明が圧倒的に強くなります。

10. よくある失敗

  1. 作ること自体が目的になる：提出できた時点で満足し、運用に使われない。
  2. 製品版数と部品版数が分かれていない：CVEが出たときに、どの製品まで影響するか追えない。
  3. 更新責任者が決まっていない：最初の1回は作れるが、その後の改訂が止まる。
  4. 機密配慮がなく、共有ルールで止まる：作っても提出判断ができず、現場が使いにくい。
  5. VEXや脆弱性対応とつながっていない：一覧表はあるが、結局OEM照会でゼロから調査する。

11. 30 / 60 / 90日で立ち上げる現実解

最初の30日

トップ3製品を対象に、Excelで最低限のSBOMを作ります。この段階では、製品版数とコンポーネント版数の対応を出せることと、担当部門・更新責任者を決めることが大切です。

次の60日

NVDなどの脆弱性情報をウォッチし、SBOMと照合する運用を始めます。OEM照会に対しては、まず「影響あり / 要確認 / 影響なし」の3段階で返せる型を作ります。

90日まで

SPDXまたはCycloneDXのどちらに寄せるかを決めます。VEX的な判断理由をテンプレ化し、OEM回答書との接続を固めます。重要なのは、“標準化した”ことよりも、脆弱性照会に一週間以内で再現可能に答えられることです。

FAQ

Q1. 自動車サプライヤーにSBOMは本当に必要ですか？

直接法令で「全サプライヤーがSBOMを作れ」と一律に書かれているわけではありません。しかし、UN R155やISO/SAE 21434の流れから、OEM要求や監査を通じてSBOM相当の管理が求められやすいのが実務上の現実です。

Q2. SPDXとCycloneDX、どちらを選ぶべきですか？

標準性やライセンス情報を重視するならSPDX、脆弱性運用やVEX連携を重視するならCycloneDXが選ばれやすいです。取引先要求と自社運用の両方で決めるのが現実的です。

Q3. Excelで始めるのは間違いですか？

間違いではありません。棚卸しの単位と更新責任が定まっていない段階ではExcelの方が現場定着しやすいことがあります。あとで標準フォーマットへ移行できるよう、項目と版管理を崩さないことが重要です。

Q4. SBOMを作ったら全部開示しないといけませんか？

いいえ。共有範囲と粒度は運用設計の問題です。社内完全版、OEM提出版、照会回答版のように分ける方が実務的です。

まとめ

SBOMは、フォーマットの名前を覚えることが本質ではありません。本質は、自社製品に何が入っているかを把握し、脆弱性が出たときに、どの製品へどの程度影響するかを説明できる状態を作ることです。

そのSBOMが、OEM照会・監査・脆弱性対応で本当に使われるか。ここに答えられるなら、そのSBOMは生きています。SBOMは提出物ではなく、説明責任を回すための実務基盤です。
